署名と公開鍵の **集約** とは、**同じメッセージに対する複数の署名を 1 つの短い署名にまとめる** ことで、メッセージの送信や検証の効率を大幅に向上させる技術です。  

### 例を見てみましょう  
次のメッセージを署名するとします。  

```bash title="メッセージ"
Per consensum ad astra
```

このメッセージに対して、秘密鍵を使って以下のような署名を作成します。  

```bash
a9c400aaf55071eedd5d151932d5cfe847ab3a5ad544521f9a3810ebb59b27c9913c86449405d3dff86c54fcc62ec48a19b182c7c87008619e88e281048f7c58b5c2cb07a1dfcf1a0e43a3678fb9efa04c218140fe5ec02d92c8eb9fbbdf312b
```

ここまでは通常の署名と同じですが、**Avalanche メインネットの Primary Network のバリデーターの大半がこのメッセージに署名するとどうなるでしょうか？**  
その場合、数百～数千の署名がこのメッセージに追加されることになります。  

例えば、1000 の署名がついた場合のデータ量を計算すると：  

- 短いメッセージ = **64 バイト**  
- 1 つの署名 = **96 バイト**  
- 1000 の署名 = **96,000 バイト**  

このような **大容量のデータ** は、ブロックチェーン上の通信の大きなボトルネックになり得ます。  

### BLS マルチシグネチャによる署名の集約  
そこで活用できるのが、**BLS マルチシグネチャの 「署名の集約」 機能** です。  

![](/common-images/cryptography/signature-aggregation.png)  

この **コンパクトな集約署名** によって、メッセージを宛先チェーンの多数のバリデーターに送信しやすくなります。**検証の際にも、この 1 つの署名をチェックするだけで済む** ため、処理が大幅に効率化されます。  

### 公開鍵の集約も必要  
署名だけでなく、**署名者の公開鍵も集約する必要があります**。  
検証時には、集約された公開鍵と集約署名を一緒に用いて、メッセージの正当性を確認します。
